JWT 101⌗ 

JSON Web Token (JWT) is a compact, URL-safe means of representing
claims to be transferred between two parties.  The claims in a JWT
are encoded as a JSON object that is used as the payload of a JSON
Web Signature (JWS) structure or as the plaintext of a JSON Web
Encryption (JWE) structure, enabling the claims to be digitally
signed or integrity protected with a Message Authentication Code
(MAC) and/or encrypted.

This is the abstract of RFC 7519, the document that brought JSON Web Tokens to life. There is a lot to unpack in this paragraph so i will explain it in a simpler way:

A JWT is a compact way of representing data that need to be transferred between two parties, the data is encoded in JSON format and it either digitally signed (with a secret code for symmetric algorithms like HMAC256) or encrypted (with a private key for asymmetric algorithms like RSA).

The signature is used to prove two things:

  • integrity: it proves that the token has not been tampered with, even if a single character has been altered then the signature attached to the token and the signature calculated would not match
  • authenticity: it proves that the token has really been sent from who it claims to be sent from, this is achieved by the fact that only the server knows the secret code so nobody else can calculate a new valid signature

If this sounds too complicated I’ll try and explain it with more details next, right now the only thing you need to know is that a JWT is just a fancy JSON object used for information exchange or user authentication.

Structure⌗

A raw JWT is formed like this:

xxxxx.yyyyy.zzzzz

There are three Base64 encoded parts separated by a dot:

  • header
  • payload
  • signature

Header⌗

The header is typically made up of two parts:

  • the token type, so JWT
  • the algorithm used for signature or encryption, for example HMAC SHA256 or RSA

Here is an example of a JWT header:

{
  "alg": "HS256",
  "typ": "JWT"
}

This JSON object is then Base64 encoded and it is placed as the first field of the JWT.

Payload⌗

The payload is the second part of the JWT and it contains the claims, claims are just informations about the user, there are three different type of claims:

  • registered claims: these claims are predifined by RFC 7519, they are not mandatory but recommended, they can be things like exp, the timestamp of token expiry or sub, the subject of this token
  • public claims: these can be defined by whoever is using the JWT, but to avoid collision they should be registered to the IANA JWT registry to avoid collisions
  • private claims: these are custom claims and they are not registered nor public claims

The minimal example of what a JWT payload should contain looks like this:

{
    "exp": 1766314986,
    "sub": "user:2315"
}

The expiry timestamp is really important because when a request with an expired JWT is sent to the server, it will generate a new one. This mechanism is needed to make sure that the attack window for a malicious third party is very short and that even if he was able to intercept a token he wouldn’t have the time to use it.

The JSON object containing the payload is Base64 encoded and added as the second field in the token.

Signature⌗

To create a signature we just need the header, payload and secret code and calculate the digital signature of that string, if we use HMAC SHA256 we would do this:

HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)

Use⌗

Let’s now see the authentication flow for a web service that uses JWTs:

  1. Client logs in with username and password
  2. The server generates the header, payload and signature
  3. The token is sent to the client, he will now store the token somewhere safe. Now that the client has aquired the JWT he can use it to authenticate himself.
  4. The client sends a request to an endpoint that needs authentication, in the HTTP request, the token will be added to the Authorization: Bearer <token> header
  5. The server receives the request and the token
  6. The server calculates the token signature and checks if the two signatures (the one sent along with the token and the one calculated right now) match, if they don’t an error is sent to the client
  7. The token is decoded by the server, if it has an expiry timestamp, the server checks if it has expired; if it has then an error is sent to the client
  8. The client has been authenticated, the request is processed and a response is sent to the client

Conclusions⌗
