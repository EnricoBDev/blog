How to handle exceptions in Java

Exceptions in Java⌗

What could possibly go wrong?⌗

In the lifetime of an application not everything will go as planned, mistakes will be made both from the user and the programmer but sometimes you will just be unlucky (a random bit flip caused by a cosmic ray will change your game state… and give a speedrunner a huge timesave)

We can categorize the sources of error in two groups:

  • internal errors, for example the user wants to do a division by zero or a method is called on a null reference
  • external errors, for example a comunication error with a remote DBMS or even an error inside the JVM, for example an “Out of memory error”

The OOP way of handling those problems is by handling them outside the normal control flow.

What even is an exception?⌗

Like everything in Java an exception is an Object that is used to tell the caller method that there is a problem and it needs to be handled.

Here is what happens in detail

  • the callee method checks conditions that can create errors
  • it throws an exception if these conditions are true
  • the caller now has to handle the exception

Let’s see an example:

public float division(int a, int b){
    if (b == 0){
        throw new IllegalArgumentException("Division by zero");
    }

    return ((float) a )/ b;
}

In this case if the divisor b is zero then an exception is thrown, now the method execution stops and we return to the caller, it can do three things:

  • prevent the exception: the caller can check beforehand if b == 0, in this case the exception will not be thrown, if the caller cannot do these checks (because it would be too complex) it can do two things:
    • it can rethrow and propagate the exception to its caller because it does not have enough context or responsibility and it doesn’t know how to handle it
    • it can catch the exception with a try-catch block and handle it

You should always try to prevent the exception, because handling exceptions can be computationally expensive and since exceptions do not follow the usual control flow it is harder to program.

The try-catch block⌗

As I said we can handle the exception by using a try-catch block, let’s see an example:

public void tryCatch(){
    try{
        int[] arr = {1, 2, 3};
        System.out.println(arr[4]);
    } 
    catch (ArrayIndexOutOfBoundsException e){
        System.out.println("Caught an array index error: ");
    }
    catch (Exception e){
        System.out.println("Caught a generic exception");
    }
}

We wrap the code that could throw an exception (in this simple example an exception will surely be thrown) in the try block, if an exception is actually thrown then the execution of the try block stops and the code inside the first catch statement that matches the exception will be executed.

The order of the catch blocks matters, if we want a detailed explanation of what happened we need to put the most generic exception type at the bottom (in this case we put the base class Exception at the bottom) because otherwise the thrown exception will always be caught with the most generic error message.

Checked and unchecked exceptions⌗

There are two big families of exceptions, checked exceptions and unchecked exceptions:

  • unchecked exceptions are tipically the result of interal errors and they are often avoidable if the programmer pays attention, also we are not obligated by the compiler to handle them
  • checked exceptions are the result of external errors and the compiler forces us to handle them (the compiler “checks” for unhandled exceptions), they are a great tool during debugging but in a production environment they shouldn’t be ever thrown or caught in a try-catch block, since they are the result of a programmer error.

We can distinguish them by looking at their superclass, checked exceptions extend Exception while unchecked exceptions extend RuntimeException.

When we throw a checked exception we need to to three things to avoid being screamed at by the compiler:

  • the exception must be declared in the called method header with the throws clause
public void close() throws IOException
  • the caller cannot ingore this exception so it either need to handle it with a try-catch block or it needs to rethrow it

(NOTE: the throws clause is compulsory for checked exceptions but it can be also used with unchecked exceptions, in this case though the compiler will not do any checks)

Gracefully restore external resources⌗

After using external resources (connection to DBMS or access to a file) we need to restore them to a normal state even when exceptions are thrown, to do this we use the finally block.

The finally block is placed after the last catch block and it is always executed:

  • it is executed if inside the try block no exceptions are thrown
  • it is executed if inside the try block an exception is thrown and it is caught
  • it is executed if inside the try block an exception is thrown and it is uncaught

Sometimes exception can be thrown even inside a finally block, for example when we are closing a file:

finally{
    try{
        if (file != null){
            file.close()
        }
    } catch (IOException e){
        System.out.println(e.printStackTrace());
    }
}

in this case there is nothing we can do, we tried our best but we couldn’t restore the resource to a normal state.

From Java 8 onwards a new construct, try-with-resources has been added, this can be used with classes that implement AutoClosable, let’s see an example:

try (FileWriter f = new FileWriter("text.txt")){
    f.write("foo");
} catch (IOException e){
    System.out.println(e.printStackTrace());
}

when this code is run, the expression inside the parenthesis is executed before the try block, if there is an IOException the catch block will handle it otherwise we just write “foo” to the file; in any case since FileWriter implements AutoClosable, the close() method will be called and any exception thrown by it will be handled automatically without using finally.
